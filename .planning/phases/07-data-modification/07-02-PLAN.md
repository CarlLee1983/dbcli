Enhance existing `src/core/data-executor.ts` with UPDATE-specific SQL building method:

**Add buildUpdateSql method to DataExecutor class:**
```typescript
private buildUpdateSql(
  tableName: string,
  data: Record<string, any>,
  whereClause: string,
  schema: TableSchema
): { sql: string, params: any[] }
```

**Implementation steps:**

1. **Parse WHERE clause into parameterized form:**
   - Extract condition from whereClause (e.g., "id=1" or "id > 5")
   - Replace literal values with placeholders
   - Build params array in order of appearance
   - Example: "id=1 AND status='active'" becomes "id=? AND status=?" with params [1, 'active']

2. **Validate all SET fields exist in schema:**
   - Check each key in data exists in table schema columns
   - Throw error: "Column 'foo' not found in table 'users'"

3. **Build SET clause with placeholders:**
   - Generate UPDATE SET col1=$1, col2=$2, ... for PostgreSQL
   - Generate UPDATE SET col1=?, col2=?, ... for MySQL
   - Order should be consistent

4. **Combine SET and WHERE with proper parameterization:**
   - SET params come first in array
   - WHERE params come after (important for $1, $2 ordering in PostgreSQL)
   - Example PostgreSQL SQL: `UPDATE users SET name=$1, email=$2 WHERE id=$3`
   - Example PostgreSQL params: ["Alice", "a@b.com", 1]

5. **Return { sql, params } for adapter.execute()**

**WHERE Clause Parsing:**
- Accept simple forms: "id=1", "id > 5", "status='active'"
- Support AND/OR: "id=1 AND status='active'"
- Extract literal values and replace with ?/$N placeholders
- This is intentionally simple — complex WHERE clauses require --dry-run verification

**Error Handling:**
- Invalid table name: thrown by adapter.execute()
- Column not found: throw descriptive error before SQL execution
- Parameterization failures: wrap with context (which field failed)

**Implementation Notes:**
- Reuse adapter-agnostic parameterization from INSERT (same pattern)
- Store both original WHERE clause and generated WHERE SQL for display
- Don't try to parse complex subqueries — require user to validate with --dry-run first

  <name>Task 2: Implement executeUpdate method in DataExecutor</name>
  <files>src/core/data-executor.ts</files>
  <action>
Add the main UPDATE execution method to DataExecutor:

**Add async executeUpdate method:**
```typescript
async executeUpdate(
  tableName: string,
  data: Record<string, any>,
  whereClause: string,
  schema: TableSchema,
  options?: DataExecutionOptions
): Promise<DataExecutionResult>
```

**Implementation steps:**

1. **Validate WHERE clause is provided and not empty:**
   - Reject if whereClause is undefined or empty string
   - Error message: "UPDATE requires --where clause (e.g., --where \"id=1\")"

2. **Enforce permission for UPDATE operation:**
   - Call `enforcePermission('UPDATE', this.permission)`
   - Catch PermissionError and re-throw with clear message (per D-05b)
   - Message: "Permission denied: Query-only mode allows SELECT only. Use Read-Write or Admin mode for UPDATE."

3. **Build UPDATE SQL via buildUpdateSql():**
   - Catch validation errors and display with context

4. **Handle --dry-run mode (per D-03c):**
   - If `options.dryRun === true`: return DataExecutionResult with status='success', rows_affected=0
   - Include generated SQL in output for verification
   - Do NOT execute query

5. **Handle --force mode (per D-03b):**
   - If `options.force === true`: skip confirmation prompt
   - Execute immediately

6. **Handle confirmation flow (per D-03a):**
   - If neither dry-run nor force: call `promptConfirmation(sql)`
   - Display generated SQL to user
   - Wait for y/n response
   - If user answers 'n': return early without executing

7. **Execute via adapter:**
   - Call `this.adapter.execute(sql, params)`
   - Capture affected row count from database response
   - Some adapters return rows array with row count; check documentation

8. **Handle errors:**
   - Catch database errors and augment with generated SQL (per D-05a)
   - Example: "Error: SQL execution failed.\nGenerated SQL: UPDATE users SET name=? WHERE id=?\nCause: [database error message]"

9. **Return DataExecutionResult:**
   - status: 'success'
   - operation: 'update'
   - rows_affected: count from database response
   - timestamp: ISO 8601 now
   - sql: generated SQL (optional verbose field)

**Error Cases:**
- WHERE clause missing or empty → throw validation error before DB call
- Permission denied → PermissionError re-thrown (caught in command handler)
- Column not found → caught by buildUpdateSql, thrown before execution
- Database execution error → augmented with SQL and re-thrown
