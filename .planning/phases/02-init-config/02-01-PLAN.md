## Key Patterns from Research

From RESEARCH.md sections on immutability, .env parsing, and config schemas:

**Immutability Rule:** All config operations use copy-on-write semantics (spread operators, no mutations).

**DATABASE_URL Parsing:** Use RFC 3986 percent-decoding via `decodeURIComponent()` for passwords with special characters (e.g., `p@$$w0rd` → `p%40%24%24w0rd` in URL).

**Component Format:** DB_HOST, DB_PORT, DB_USER, DB_PASSWORD, DB_NAME, DB_SYSTEM as environment variables.

**Validation:** Use Zod for schema-first validation catching errors at read time, not runtime failures.

**Error Handling:** Custom error classes (EnvParseError, ConfigError) with context for debugging.

**Database Defaults:** PostgreSQL port 5432, MySQL/MariaDB 3306. Store in adapters/defaults.ts for per-system config.

## From Phase 1 Summary

- TypeScript 5.9.3 with strict mode enforced
- Vitest 1.6.1 with node environment configured
- No console.log allowed; use structured errors instead
- Bun.file preferred over fs for file I/O

   - If file doesn't exist, return default config: { connection: { system: 'postgresql', host: 'localhost', port: 5432, user: '', password: '', database: '' }, permission: 'query-only' }
   - If file exists, parse JSON and validate against DbcliConfigSchema
   - Throw ConfigError if JSON is invalid or validation fails (with error message from Zod)

2. **validate(raw: unknown): DbcliConfig**
   - Call DbcliConfigSchema.parse(raw) and return result
   - Catch Zod error and throw ConfigError with descriptive message
   - Used to validate user-provided config in write() and by tests

3. **merge(existing: DbcliConfig, updates: Partial<DbcliConfig>): DbcliConfig**
   - IMMUTABILITY CRITICAL: Never modify existing or updates
   - Return new object with spread operator: `{ ...existing, ...updates, connection: { ...existing.connection, ...(updates.connection || {}) }, metadata: { ...existing.metadata, ...(updates.metadata || {}), createdAt: existing.metadata?.createdAt || new Date().toISOString() } }`
   - This pattern preserves nested immutability
   - Return type is full DbcliConfig, validated

4. **async write(path: string, config: DbcliConfig): Promise<void>**
   - Validate config via this.validate(config) first (fail early if invalid)
   - Stringify with JSON.stringify(config, null, 2) for readable output
   - Use Bun.file(path).write(json) to write
   - Throw ConfigError if write fails

Reference RESEARCH.md Pattern 1 for exact implementation (sections: "Core config module: immutable operations" and "Usage: Read → Validate → Merge → Write").

Error messages must be user-facing (reference RESEARCH.md Pitfall 3 for .dbcli overwrite scenario error message).
