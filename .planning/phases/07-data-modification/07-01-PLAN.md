Create `src/core/data-executor.ts` with DataExecutor class for executing INSERT/UPDATE with permission checks, parameterized queries, and error handling:

**Core Responsibilities:**
1. Permission enforcement via `enforcePermission()` for INSERT/UPDATE operations
2. Build parameterized SQL queries with placeholders ($1, $2 for PostgreSQL; ? for MySQL)
3. Execute with parameter binding to prevent SQL injection
4. Handle pre-execution confirmation flow
5. Support --dry-run and --force modes
6. Return DataExecutionResult with row count and metadata

**Constructor:**
```typescript
constructor(
  private adapter: DatabaseAdapter,
  private permission: Permission
)
```

**Key Methods:**

1. **buildInsertSql(tableName: string, data: Record<string, any>, schema: TableSchema): { sql: string, params: any[] }**
   - Validate all fields exist in table schema
   - Generate parameterized INSERT statement
   - Return both SQL and params array
   - Example output for PostgreSQL:
     - SQL: `INSERT INTO users (name, email) VALUES ($1, $2)`
     - params: ["Alice", "a@b.com"]
   - Example output for MySQL:
     - SQL: `INSERT INTO users (name, email) VALUES (?, ?)`
     - params: ["Alice", "a@b.com"]

2. **async executeInsert(tableName: string, data: Record<string, any>, schema: TableSchema, options?: DataExecutionOptions): Promise<DataExecutionResult>**
   - Call `enforcePermission('INSERT', this.permission)` — throw PermissionError if rejected
   - Build SQL via buildInsertSql()
   - If dryRun: return result with `rows_affected: 0` and SQL shown
   - If not force: call `promptConfirmation(sql)` with user y/n prompt
   - Execute `adapter.execute(sql, params)`
   - Catch errors and add SQL to error message (per D-05a)
   - Return DataExecutionResult with rows_affected from database response

**Error Handling:**
- Catch PermissionError and re-throw with clear message (per D-05b): "Permission denied: Query-only mode allows SELECT only. Use Read-Write or Admin mode for INSERT."
- Catch database errors and include generated SQL in message
- Handle NULL/undefined values gracefully (map to null for database)

**Implementation Notes:**
- Use same permission checking pattern as QueryExecutor (import enforcePermission)
- Adapter.execute() already supports parameterized queries — pass sql and params
- For MySQL/MariaDB, use ? placeholders; for PostgreSQL, use $1, $2, etc.
- Store generated SQL for display and error reporting

Create new file `src/commands/insert.ts` with insert command handler following query.ts pattern:

**Function signature:**
```typescript
export async function insertCommand(
  table: string,
  options: {
    data?: string        // optional: JSON for single row (--data flag)
    dryRun?: boolean     // --dry-run flag
    force?: boolean      // --force flag
  }
): Promise<void>
```

**Implementation structure (follow query.ts pattern):**

1. **Argument validation:**
   - Validate table name is not empty
   - Error: "Table name required"

2. **Data input handling:**
   - Primary method: Read from stdin if available
   - Alternative: Use --data flag for simple single-row inserts
   - If both stdin and --data provided, use stdin
   - If neither: error "No data provided. Use stdin or --data flag"

3. **Parse JSON data:**
   - Parse JSON from stdin or --data flag
   - Validate it's an object or array of objects
   - Singular: convert `{obj}` to `[{obj}]` for processing
   - Error format: "Invalid JSON in input: {error message}"
   - Result: array of Record<string, any>

4. **Load configuration:**
   - Read `.dbcli` config file
   - Validate connection exists: "Run \"dbcli init\" first"

5. **Create database adapter:**
   - Use AdapterFactory.createAdapter(config.connection)
   - Call adapter.connect()

6. **Get table schema:**
   - Call adapter.getTableSchema(table)
   - Catch errors: "Table '{table}' not found"

7. **Process each row (typically 1 for insert):**
   - For first row only (Phase 7 is single-insert; batch deferred to Phase 8):
     - Create DataExecutor: `new DataExecutor(adapter, config.permission)`
     - Call: `executor.executeInsert(table, data[0], schema, { dryRun: options.dryRun, force: options.force })`

8. **Format output as JSON:**
   - Use result from executeInsert() directly
   - Serialize to JSON and console.log

9. **Error handling chain:**
   - PermissionError: Display "❌ Permission Denied" with operation type and required permission level
   - ConnectionError: Display "❌ Database Connection Failed" with message
   - ValidationError (JSON parse, missing table): Display "❌ Insert Error" with message
   - Other errors: Display with context

10. **Exit codes:**
    - Success: exit 0 (implicit)
    - Any error: exit 1

**JSON Output Format:**
- Success: `{"status":"success","operation":"insert","rows_affected":1,"timestamp":"2026-03-25T10:30:00Z"}`
- Dry-run: same but rows_affected=0 and sql field included for verification

**Example CLI usage:**
```bash
# Primary: JSON stdin
echo '{"name":"Alice","email":"a@b.com"}' | dbcli insert users

# Alternative: --data flag
dbcli insert users --data '{"name":"Alice","email":"a@b.com"}'

# With confirmation
# (shows SQL, waits for y/n)

# Force skip confirmation
dbcli insert users --data '{"name":"Alice"}' --force

# Dry-run (show SQL without executing)
dbcli insert users --data '{"name":"Alice"}' --dry-run
```

**Implementation Notes:**
- Follow exact structure and error handling pattern from src/commands/query.ts
- Use same permission error messages and formatting
- Keep function focused: validation → config → adapter → execute → output
- Connection cleanup in finally block: `await adapter.disconnect()`
- Stdin handling: use process.stdin in async/readable stream mode
