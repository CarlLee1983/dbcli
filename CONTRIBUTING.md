# Contributing to dbcli

Thank you for your interest in contributing to dbcli! This guide explains how to set up your development environment and submit changes.

## Code of Conduct

Be respectful, inclusive, and constructive in all interactions.

---

## Development Setup

### Prerequisites

- **Bun 1.3.3+** — Bun runtime and package manager
  - Install: https://bun.sh/docs/installation
  - Verify: `bun --version` should show >= 1.3.3

- **Node.js 18.0.0+** — For compatibility with dbcli's declared engines
  - Install: https://nodejs.org/
  - Verify: `node --version`

- **PostgreSQL 12+ or MySQL 8.0+** — For testing with real databases
  - Optional for unit tests; required for integration tests

- **Git** — For version control

### Clone Repository

```bash
git clone https://github.com/yourusername/dbcli.git
cd dbcli
```

### Install Dependencies

```bash
bun install
```

This installs devDependencies and runtime dependencies in ~10 seconds.

### Verify Setup

```bash
# Run tests
bun test --run

# Build CLI
bun run build

# Test executable
./dist/cli.mjs --help
```

All commands should succeed without errors.

## Development Workflow

### 1. Create Feature Branch

```bash
git checkout -b feature/your-feature-name
```

Branch naming: `feature/`, `fix/`, `docs/`, `test/`, `refactor/`

### 2. Make Changes

Edit source files in `src/`. Follow these guidelines:

#### Code Style

- **Immutability:** NEVER mutate objects; use spread operators for updates
  ```typescript
  // WRONG: mutation
  user.name = "Alice"

  // CORRECT: immutability
  const updated = { ...user, name: "Alice" }
  ```

- **Error Handling:** Always handle errors comprehensively
  ```typescript
  try {
    const result = await riskyOperation()
    return result
  } catch (error) {
    console.error('Operation failed:', error)
    throw new Error('User-friendly message')
  }
  ```

- **Input Validation:** Validate all user input with Zod
  ```typescript
  import { z } from 'zod'

  const schema = z.object({
    email: z.string().email(),
    age: z.number().int().min(0).max(150)
  })

  const validated = schema.parse(input)
  ```

- **File Size:** Keep files under 800 lines; extract utilities as needed

- **Function Size:** Functions should be < 50 lines; break up larger functions

- **No Console:** Avoid `console.log()` in production code (OK for debugging during development, remove before commit)

#### TypeScript

- Strict mode enabled in `tsconfig.json`
- No `any` types; use generics or union types
- Explicit return types for public functions

#### Testing

- Write tests BEFORE implementing features (TDD)
- Aim for 80%+ code coverage
- Unit tests for pure functions
- Integration tests for database operations
- Performance tests for critical paths (see vitest.config.ts)

### 3. Write Tests

```bash
# Add test file alongside source
src/commands/mycommand.ts
tests/unit/commands/mycommand.test.ts

# Write test first (RED)
import { test, expect } from 'vitest'
import { myCommand } from '../../src/commands/mycommand'

test('myCommand returns expected value', () => {
  const result = myCommand()
  expect(result).toEqual({ ... })
})

# Run test (should FAIL)
bun test --run

# Implement (GREEN)
# ... code changes ...

# Test again (should PASS)
bun test --run

# Refactor if needed (IMPROVE)
```

### 4. Run Tests

```bash
# Quick run (unit + integration, no bench)
bun test --run

# Full suite with performance benchmarks
bun test --run --bench

# Watch mode (re-run on file changes)
bun test
```

### 5. Lint and Format

```bash
# Check code style
bun run lint

# Auto-fix style issues
bun run lint:fix

# Format code (Prettier)
bun run format
```

### 6. Build and Verify

```bash
# Build CLI
bun run build

# Verify executable
./dist/cli.mjs --help
./dist/cli.mjs --version

# Verify binary size
ls -lh dist/cli.mjs  # Should be 1.0-1.5MB
```

### 7. Commit

Follow [Conventional Commits](https://www.conventionalcommits.org/) format:

```
<type>: <description>

<optional body>
```

Types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `chore`

Examples:
```bash
git commit -m "feat: [commands] add --dry-run flag to update command"
git commit -m "fix: [permission-guard] handle CTE in WHERE clause"
git commit -m "docs: [readme] add troubleshooting section"
git commit -m "test: [permission-guard] add 5 tests for edge cases"
```

### 8. Push and Create Pull Request

```bash
git push origin feature/your-feature-name
```

Create PR on GitHub. Include:
- Description of changes
- Why the change is needed
- Testing done
- Breaking changes (if any)

## Testing

### Test Structure

```
tests/
├── unit/              # Pure function tests (no I/O)
├── integration/       # Database and CLI tests
└── perf/              # Performance benchmarks
```

### Running Tests

```bash
# Unit tests only
bun test --run --grep "unit"

# Integration tests only (requires test database)
bun test --run --grep "integration"

# Performance benchmarks
bun test --run --bench

# Coverage report
bun test --run --coverage
```

### Test Database Setup

Integration tests expect test databases:

**PostgreSQL:**
```bash
# Create test database
createdb dbcli_test

# Verify
psql dbcli_test -c "SELECT 1"
```

**MySQL:**
```bash
# Create test database
mysql -u root -p -e "CREATE DATABASE dbcli_test"

# Verify
mysql -u root -p dbcli_test -e "SELECT 1"
```

Set environment variables in `.env.test`:
```
TEST_DATABASE_URL=postgresql://localhost/dbcli_test
# OR
TEST_DATABASE_URL=mysql://root@localhost/dbcli_test
```

### Code Coverage

```bash
bun test --run --coverage

# View coverage report
open coverage/index.html  # macOS
xdg-open coverage/index.html  # Linux
start coverage/index.html  # Windows
```

Target: 80%+ coverage on src/ directory.

## Release Process

### Version Numbering

Follow [Semantic Versioning](https://semver.org/):
- `MAJOR` — Breaking changes
- `MINOR` — New features (backward compatible)
- `PATCH` — Bug fixes (backward compatible)

### Pre-Release Checklist

1. **Update version in package.json:**
   ```bash
   npm version patch  # 1.0.0 → 1.0.1
   npm version minor  # 1.0.0 → 1.1.0
   npm version major  # 1.0.0 → 2.0.0
   ```

2. **Update CHANGELOG.md:**
   - Add new `## [X.Y.Z] - YYYY-MM-DD` section
   - List new features, fixes, known issues
   - Use [Keep a Changelog](https://keepachangelog.com/) format

3. **Verify build:**
   ```bash
   bun run build
   ls -lh dist/cli.mjs  # Check size < 1.5MB
   ```

4. **Run full test suite:**
   ```bash
   bun test --run --bench
   ```

5. **Test across platforms:**
   ```bash
   # Push to GitHub, verify GitHub Actions passes all matrix tests
   git push origin main
   # Wait for CI to pass on ubuntu-latest, macos-latest, windows-latest
   ```

### Publishing to npm

```bash
# Ensure you're logged in to npm
npm login

# Publish (prepublishOnly hook will build)
npm publish

# Verify on npm registry
npm info dbcli
```

The `prepublishOnly` hook in package.json automatically runs `bun run build` before publishing, ensuring a fresh binary is uploaded.

### Post-Release

1. Verify installation works:
   ```bash
   npm install -g dbcli@X.Y.Z
   dbcli --version
   ```

2. Test zero-install:
   ```bash
   cd /tmp
   npx dbcli@X.Y.Z --help
   ```

3. Update GitHub releases page with release notes from CHANGELOG.md

## Troubleshooting

### "bun: command not found"

Install Bun: https://bun.sh/docs/installation

### "TypeError: Cannot find module 'zod'"

Run: `bun install`

### Tests fail with database errors

Ensure test databases are created and TEST_DATABASE_URL is set in `.env.test`

### Build fails with TypeScript errors

Run `bun test --run` to see full error messages and stack traces

### Windows: "chmod: command not found" in build

This is expected. The build script includes `|| true` fallback. Windows doesn't have chmod; npm's .cmd wrapper handles executable bit.
