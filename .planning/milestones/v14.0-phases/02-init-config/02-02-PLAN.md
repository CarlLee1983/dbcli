## Key Patterns from Research

**Interactive Prompts:** Use @inquirer/prompts with fallback to minimal synchronous prompts if Bun compatibility issues arise.

**Hybrid Initialization:** Read .env first, then prompt for missing values. Only prompt for values not found in environment.

**Config Merge Flow:** Load existing config → validate → merge new values → write back.

**Overwrite Protection:** Before writing .dbcli, check if file exists; if so, prompt user for confirmation.

**Non-Interactive Mode:** Support `--no-interactive` flag for CI/automation scenarios.

Create src/utils/prompts.ts with a promptUser object exporting three async functions with @inquirer/prompts and fallback support:

**Design Pattern (from RESEARCH.md Pattern 3):**
Each function tries to import and use @inquirer/prompts. If import fails (Bun compatibility issue), fall back to synchronous prompts using console.log + readlineSync (from bun:readline or built-in Bun.stdin).

1. **async promptUser.text(message: string, defaultValue?: string): Promise<string>**
   - Try: Import text from @inquirer/prompts, call text({ message, default: defaultValue })
   - Fallback: console.log the message, readlineSync.question('> '), return answer or defaultValue
   - Handle empty input by returning defaultValue if provided
   - Example usage: `await promptUser.text('Database host:', 'localhost')`

2. **async promptUser.select(message: string, choices: string[]): Promise<string>**
   - Try: Import select from @inquirer/prompts, call select({ message, choices })
   - Fallback: console.log message, print numbered choices, readlineSync.question('> '), return choices[index]
   - Handle invalid input by returning first choice (or prompt again)
   - Example usage: `await promptUser.select('Database system:', ['postgresql', 'mysql', 'mariadb'])`

3. **async promptUser.confirm(message: string): Promise<boolean>**
   - Try: Import confirm from @inquirer/prompts, call confirm({ message })
   - Fallback: console.log(message + ' (y/n) '), readlineSync.question('> '), return answer.toLowerCase() === 'y'
   - Example usage: `await promptUser.confirm('Overwrite existing .dbcli?')`

**Implementation Details:**
- Use dynamic imports (await import()) to avoid hard dependencies if @inquirer/prompts has issues
- Catch any error during import or execution; if caught, run fallback
- For readline.sync, use Bun built-in: `import { stdin } from 'bun'` or use `console.log` + a simple fallback
- RESEARCH.md notes Bun compatibility may have stdin EPERM errors; defensive fallback design handles this
- No error throwing in fallback; gracefully degrade to simpler UX

Reference RESEARCH.md Pattern 3 code examples for exact implementation.

**Testing Note:** Unit tests (not in this plan) can substitute this module with mock prompts easily due to isolated design.

Create src/commands/init.ts with a Commander.js command object implementing the init flow:

**Command Definition:**
- Command name: 'init'
- Description: 'Initialize dbcli configuration with .env parsing and interactive prompts'
- Options:
  - --no-interactive: Non-interactive mode (requires all flags, no prompts)
  - --host <host>: Database host
  - --port <port>: Database port
  - --user <user>: Database user
  - --password <password>: Database password
  - --name <name>: Database name
  - --system <system>: Database system (postgresql, mysql, mariadb)
  - --force: Skip overwrite confirmation if .dbcli exists

**Initialization Flow (Pseudo-code from RESEARCH.md Code Examples):**

```
1. Load existing .dbcli if it exists via configModule.read()
2. Try to parse .env via parseEnvDatabase(process.env)
3. Determine database system:
   - Use --system option if provided
   - Use parsed .env system if available
   - Use CLI options if provided
   - Otherwise: prompt user to select (postgresql / mysql / mariadb)

4. For each connection parameter (host, port, user, password, database):
   - Check if provided via CLI option
   - Check if available from .env
   - Apply database defaults (via getDefaultsForSystem)
   - Prompt user if value is missing

5. Ask user to select permission level:
   - Default: 'query-only'
   - Options: query-only / read-write / admin

6. Merge new values into config via configModule.merge()

7. Before writing, check if .dbcli already exists:
   - If it exists and --force not provided: prompt "File .dbcli already exists. (o)verwrite/(q)uit? "
   - If quit selected, exit cleanly (no error)
   - If overwrite, proceed to write

8. Write config to .dbcli via configModule.write()
9. Print success message: "✓ Configuration saved to .dbcli"
```

**Implementation Details from RESEARCH.md:**
- Use configModule.read() to get existing config (returns default if file doesn't exist)
- Use parseEnvDatabase(process.env) to detect .env variables
- For each missing value, use promptUser.text/select/confirm
- Show defaults in prompts: `await promptUser.text('Database host:', defaults.host)`
- Validate port as integer before merging: `parseInt(portString, 10)`
- Handle non-interactive mode: if --no-interactive set but required values missing, throw error with hint
- Error messages should reference RESEARCH.md Pitfall sections:
  - Pitfall 1 (percent-encoding): "PASSWORD may need percent-encoding; see `dbcli init --help`"
  - Pitfall 3 (overwrite protection): "File .dbcli already exists. (o)verwrite/(q)uit?"

**Exports:**
- `export const initCommand = new Command('init')...`
